116 /* C has some very silly default behaviours. One of these is to define all
117  * symbols declared at file level as global in scope (i.e. visible to the
118  * linker). This is a recipe for trouble. Secondly, C overloads the 'static'
119  * storage class (which defines how data is allocated to memory) with a second
120  * completely independent meaning - that the scope of the symbol name will be
121  * restricted to the current C source file only. While this is behaviour that we
122  * want, the use of the word 'static' is odd, to say the least. So, use the
123  * following definitions to discipline yourself to sensible behaviour - define
124  * truly global variables and functions as 'GLOBAL', and those which are to be
125  * visible only in the current C source file as 'LOCAL'.
126  * [Each global function definition should be matched with a corresponding
127  * function prototype in the corresponding header file, and each global
128  * variable definition should be matched with a corresponding 'extern'
129  * declaration in that same header file. There should be no 'extern'
130  * declarations outside of header files.] */
131 #define GLOBAL
132 #define LOCAL static
133 
134 /* Another odd design decision was the requirement to insert the keyword "break"
135  * after every element of a switch-case construct if you did not want to run on
136  * and execute the code of the next element. That is, most of the time you must
137  * insert the word "break", and only occasionally do you write code in which you
138  * don't want it. So on average, if a "break" is missing in a case statement, it
139  * is a bug. So, on those rare occasions when we do intend to run on, we want to
140  * let people know that, this time, it isn't a bug. */
141 #define NOBREAK /* Show reader that 'break' was not omitted by accident */
142 #define FALLTHROUGH /* or if you prefer, use this word instead. */

270 /* Finally, one more piece of assertion checking that is particularly relevant
271  * to C code. Unlike C++, C does not allow passing parameters to functions by
272  * reference. It allows passing by address/by pointer, but this is not exactly
273  * the same thing. It is quite possible to pass NULL to any parameter in C that
274  * is expecting an address or pointer, and the compiler will not object. After
275  * all, sometimes functions check for NULL pointers and take different actions
276  * if one is passed. But all too often, functions cannot handle getting a NULL
277  * pointer, and will crash if one is passed. An assertion check helps here.
278  * Uncomment the following line if you want to perform NULL checking.
279  * Comment it out if you want to suppress NULL checking. */
